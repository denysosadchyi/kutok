# Ключові екрани головного flow — замовлення на wireframes

> Крок 07. Виокремлено екрани **ГОЛОВНОГО flow** — MAIN JOB (бік Ані-шукача) зі [flows.md](../flows.md):
> наскрізний шлях головної персони (шукач → куди заїхати), що закриває головний job —
> *безпечно знайти, з ким і де жити*.
>
> **Що сюди НЕ потрапило (розкотимо кроком 08):** бік господаря (E, Олена), інфраструктурні flow
> (G1 вхід, G3 скарга/блок), RJ4/RJ5 окремо, групи A (приватність, заблоковані, акаунт), G–H (право, підтримка).
>
> **Правила відбору:** беремо лише вузли-**екрани** MAIN-flow. Стани (`empty/loading/error`) і дії-кнопки
> («Хочу познайомитись») — **не екрани** (так у sitemap). Кожен екран має і **job** (jtbd), і **місце у flow** (flows) —
> інакше не береться. Назви — точно зі [sitemap.md](../sitemap.md).
>
> **Стани:** ✓ = стан реальний у сценарії · «—» = сценарій його не дає.

## Розшифровка по екранах

### 1. Створення профілю
- **Job:** SJ1 — *«Коли інша сторона дивиться на мене як на кандидата, я хочу виглядати надійною і реальною людиною, щоб мені відповідали…»*; також SJ2.
- **Місце у flow:** MAIN — перший вузол `Create` після старту («Аня заходить уперше» → Створення профілю → Стрічка). Сюди ж веде G1 для нового номера.
- **Стани:**
  - **Успіх ✓** — профіль збережено → стрічка (`Create → Feed`).
  - **Завантаження ✓** — збереження + аплоуд фото (фото — обов'язкове поле профілю).
  - **Помилка ✓** — збій аплоуду фото / валідації (дзеркало `UpErr` «збій аплоуду фото» з flow E).
  - **Порожній —** — це форма вводу; стану «нема даних для показу» сценарій не дає.

### 2. Стрічка / каталог: кімнати + люди, з фільтром
- **Job:** RJ1 — *«Коли я починаю шукати серед десятків оголошень і профілів, я хочу швидко відсіяти тих, хто точно не підходить, щоб не витрачати увагу на нерелевантне.»*
- **Місце у flow:** MAIN — вузол `Feed`; повністю розгорнутий у RJ1.
- **Стани:**
  - **Порожній ✓** — ядровий стан цього екрана, причому **два різні**: фільтр завузький (`E`) і cold-start, база порожня (`ZI`) — RJ1 розводить їх явно.
  - **Завантаження ✓** — RJ1 `L` «оновлюємо результати».
  - **Успіх ✓** — є картки → перехід у перегляд профілю/оголошення.
  - **Помилка ✓ ¹** — *перевернуто з «—» на крок побудови станів.* Початково сценарій моделював лише «порожньо» (контент), а не технічний збій завантаження — вузла-помилки стрічки flows.md не дає. Додано свідомим рішенням: будь-яке завантаження каталогу може впасти (нема мережі) → екран `listings-error.html` з виходом «Спробувати ще». Retry-патерн запозичено з `NetErr → повторити` (MAIN/RJ3), бо власного вузла flows.md не має — кандидат дорисувати у flows.md окремим вузлом.

### 3. Профіль іншої людини
- **Job:** RJ2 — *«Коли я натрапив(ла) на профіль, що здається підходящим, я хочу переконатися що це реальна людина, щоб не ризикувати наодинці з незнайомцем.»*; також RJ4.
- **Місце у flow:** MAIN — вузол `View`; повністю у RJ2. Тут живе дія-кнопка «Хочу познайомитись» (стан картки, не екран).
- **Стани:**
  - **Порожній ✓** — RJ2 явно: «Профіль заповнений? ні, порожній» (`Stuck2`) + «ще немає відгуків (cold start)» (`G`).
  - **Завантаження ✓** — підвантаження профілю/бейджа/відгуків; плюс стан «надсилаємо» кнопки інтересу.
  - **Помилка ✓** — збій надсилання «Хочу познайомитись» (`помилка мережі`, RJ3 `Err` / MAIN `NetErr`) — це стан кнопки на цьому екрані.
  - **Успіх ✓** — повний профіль із бейджем (готова до контакту) + «інтерес надіслано».

### 4. Оголошення кімнати
- **Job:** RJ2 / RJ4 — той самий «переконатися до контакту», застосований до житла; закриває «**де** жити» з main job.
- **Місце у flow:** MAIN — друга гілка вузла `View` («Профіль іншої людини / Оголошення кімнати»). Тут теж є дія «Хочу познайомитись».
- **Стани:**
  - **Завантаження ✓** — підвантаження фото/опису/умов.
  - **Помилка ✓** — збій надсилання інтересу (та сама кнопка-дія, що й на профілі).
  - **Успіх ✓** — оголошення показано, можна діяти.
  - **Порожній —** — опубліковане оголошення завжди має обов'язковий контент (фото/ціна/район), у кімнати нема відгуків → стану «нема даних» сценарій не дає.

### 5. Верифікація: телефон + селфі
- **Job:** Main Job — *«…бути впевненою що людина реальна…»* (стати перевіреною, не лише бачити перевірених); RJ2 (тут формується мій власний бейдж).
- **Місце у flow:** MAIN — гейт `Q3` «Я верифікована?» → `Verify` (спрацьовує на дії «Хочу познайомитись», не на вході). Детально розгорнуто у G2.
- **Стани:**
  - **Завантаження ✓** — G2 явно: «надсилаємо SMS», «перевіряємо селфі».
  - **Помилка ✓** — найбагатший на помилки екран: SMS не дійшла / код невірний / нема дозволу камери / фото не збіглось / лайвнес не пройдено.
  - **Успіх ✓** — бейдж `телефон ✓ / фото ✓` — дію розблоковано (`Done`).
  - **Порожній —** — покроковий процес; стану «нема даних» нема.

### 6. Мої заявки: надіслані / очікують
- **Job:** RJ3 — *«Коли я вирішив(ла) написати комусь першим, я хочу знати що людина навпроти теж відкрита, щоб не відчувати що вторгаюся.»*
- **Місце у flow:** MAIN — вузол `Wait` «чекаємо відповідь»; у RJ3 це `My` — «дім стану чекаю» (надіслані / очікують).
- **Стани:**
  - **Завантаження ✓** — `Wait` / `My` (loading) «чекаємо відповідь».
  - **Порожній ✓** — ще не надіслано жодної заявки.
  - **Успіх ✓** — список зі статусами (надіслано / взаємно / відхилено); «взаємно» розблоковує чат. *(Відхилено / тиша — це контент усередині списку, не окремий стан-екран.)*
  - **Помилка —** — збій надсилання живе на картці профілю/оголошення (екрани 3–4), не тут; завантаження самого списку сценарій помилкою не моделює.

### 7. Чат (текст)
- **Job:** RJ3 (розмова почалась лише за взаємного інтересу) + RJ4 (лінк на профіль співрозмовника — готовність до зустрічі).
- **Місце у flow:** MAIN — вузол `Chat`; у RJ3 це `G` «Чат — розблоковано».
- **Стани:**
  - **Порожній ✓** — RJ3 `GEmpty` «ще немає повідомлень — почніть розмову».
  - **Завантаження ✓** — підвантаження історії повідомлень.
  - **Успіх ✓** — «розмова почалась» (`Ok`).
  - **Помилка —** — базовий текст-чат MVP; сценарій моделює лише empty/success, вузла збою надсилання повідомлення не дає.
