# Information Architecture — Куток

> Загальна навігація **продукту** (мобільного застосунку), не воркспейсу.
> Зібрано з усіх екранів [sitemap.md](./sitemap.md) (групи A–F) і шляхів [flows.md](./flows.md).
> Правило: вузли-екрани існують у sitemap; нових екранів не введено. Контейнери навігації (онбординг, вкладки) — позначено окремо.

---

## Навігаційна модель

Mobile-first. Три рівні навігації:

1. **Онбординг (перший запуск)** — реєстрація + верифікація. Обов'язковий до першого *контакту*, **не до перегляду** (lazy verification, R1): можна одразу ввійти в таб-бар і гортати стрічку, а верифікація спрацьовує на дії «Хочу познайомитись». Поза постійною навігацією.
2. **Таб-бар — 4 вкладки** — постійна навігація застосунку: **Пошук · Заявки · Чати · Профіль**. Розбито за дієсловами людини, а не за «розділами».
3. **Деталі / дії всередині вкладки** — профілі, оголошення, чат, редагування. Сюди ж — дія «Хочу познайомитись».

**Роль — не окрема навігація, а накладка.** Одна людина може бути і шукачем, і господарем. Тому немає окремого «режиму господаря»: його екрани (розмістити кімнату, мої оголошення) живуть усередині вкладки **Профіль**, а вхідні заявки на його кімнату — у вкладці **Заявки**, спільній для обох ролей.

**Стани ≠ навігація.** empty / loading / error (з [flows.md](./flows.md)) — стани екранів, не пункти навігації. У дереві їх немає.

---

## Єдине дерево навігації

```
Куток · мобільний застосунок
│
├─ ⓿ Онбординг ……………………………………… [потік, до таб-бару] · [P][S]
│   ├─ Вхід (existing-user): телефон → SMS-код …… повернення / новий пристрій — не лише перший запуск
│   ├─ Створення профілю ……………………………… (SJ1, SJ2) · точка входу
│   ├─ Згода на дані + біометрію · Умови / Політика … (правовий обов'язок — група G у sitemap)
│   └─ Верифікація: телефон + селфі ………………… (Main, RJ2) · під-кроки: телефон → SMS-код → селфі/лайвнес → обробка → (N невдач) ескалація/підтримка
│
└─ ▣ Таб-бар (4 вкладки) ……………………………… [постійна навігація]
    │
    ├─ ① Пошук ……………………………………………… [вкладка] · [P] (шукач)
    │   ├─ Стрічка / каталог: кімнати + люди, з фільтром … (RJ1)
    │   │     └─ фільтр (район, ціна, стать) … панель, стан — не екран
    │   ├─ Профіль іншої людини ………………………… (RJ2, RJ4)
    │   │     ├─ дія «Хочу познайомитись» → надсилає Інтерес … (RJ3)
    │   │     └─ дія «Поскаржитись/заблокувати» → форма скарги … (безпека; Main/EJ2)
    │   └─ Оголошення кімнати ………………………… (RJ2, RJ4)
    │         └─ дія «Хочу познайомитись» → надсилає Інтерес … (RJ3)
    │
    ├─ ② Заявки  ·  лічильник-бейдж нових (in-app) ……… [вкладка] · [P][S]
    │   ├─ Мої заявки: надіслані / очікують ………………… (RJ3 — дім стану «чекаю»; відкликати)  [P]
    │   └─ Вхідні заявки / кандидати …………………… (RJ3)
    │
    ├─ ③ Чати  ·  лічильник-бейдж непрочитаних (in-app) … [вкладка] · [P][S]
    │   ├─ Список діалогів ……………………………… (RJ3)
    │   └─ Чат (текст) · лінк на профіль співрозмовника … (RJ3, RJ4) · відкритий лише при взаємності
    │         └─ дія «Підтвердити заселення» → стартує вікно відгуку 14 днів … (якір RJ5; безпека #6)
    │
    └─ ④ Профіль / Кабінет ………………………………… [вкладка] · [P][S]
        ├─ Мій профіль — перегляд і редагування …… (SJ1, RJ4)
        │     └─ бейдж верифікації, повнота, відгуки про мене
        ├─ Мої оголошення · статуси: активне/призупинено/«зайнято»/архів … (RJ1, supply) · [S] господар
        │     └─ Створення / редагування оголошення … (RJ1) · [S] господар
        ├─ Налаштування приватності: видимість person-card / blur фото … (безпека #1)
        ├─ Заблоковані: список + розблокувати … (безпека)
        ├─ Керування акаунтом: вихід / видалення акаунта і даних … (trust + правовий обов'язок)
        ├─ Підтримка / звернення · Умови / Політика приватності … (інфраструктура/право — група G)
        ├─ Відновлення доступу (втрата номера) … (phone-auth recovery — група G)
        └─ Лишити відгук про людину …⏳ post-MVP …… (RJ5)
```

**Легенда:** `[P]` потрібен primary (Аня, шукач) · `[S]` secondary (Олена-господар / Артем) · `⏳` post-MVP · `(RJx)` job зі [jtbd.md](./research/jtbd.md) · «група G/H» — інфраструктурні/відкладені екрани зі [sitemap.md](./sitemap.md).

## Як шляхи (flows) лягають на навігацію

| Flow (job) | Стартова вкладка | Перетин вкладок | Діаграма у [flows.md](./flows.md) |
|---|---|---|---|
| **Main Job** | Онбординг → ① Пошук | ① → (дія) → ② Заявки → ③ Чати | ✅ |
| **RJ1** — відсіяти | ① Пошук | лишається в ① | ✅ |
| **RJ2** — довіра до контакту | ① Пошук → Профіль/Оголошення | лишається в ① | ✅ |
| **RJ3** — взаємний інтерес | ① Пошук (дія) → ② Заявки → ③ Чати | три вкладки послідовно | ✅ |
| **E — бік господаря** | ④ Профіль (оголошення) → ② Заявки → ③ Чати | публікація → кандидати → чат | ✅ |
| **RJ4** — підготовка до зустрічі | ③ Чати → лінк на Профіль | у межах ③ / ① | ⬜ покриває екран профілю |
| **RJ5** — відгук | ④ Профіль | лишається в ④ | ⬜ post-MVP |

Ключовий продуктовий перехід **① → ② → ③** (побачив → інтерес взаємний → чат) — це і є кістяк trust-gate. Він навмисно проходить через три різні вкладки, щоб кожен крок був видимою, окремою дією.

## Рішення і відкриті питання

- **Чому 4 вкладки, а не 3.** Заявки (② — намір, ще не розблоковано) і Чати (③ — розблокована розмова) можна злити в один «Inbox» з двома підтабами. Розділив свідомо: для trust-продукту різниця «заявка vs відкритий діалог» — змістова, не технічна. Об'єднання — кандидат на пост-валідацію після інтерв'ю.
- **Господар без окремого режиму.** Якщо інтерв'ю покаже, що primary — господар (sitemap/jtbd: ринок supply-обмежений), «Розмістити кімнату» підніметься з ④ у помітнішу дію (глобальний `+` або власна вкладка). Поки — під ④.
- **Чого немає в навігації навмисно** (з sitemap «Під питанням»): збережені/обране, скоринг сумісності, соц-граф. *Push*-сповіщення — поза MVP, але **in-app лічильники** на ② і ③ потрібні. Керування акаунтом (вихід / видалення даних) додано у ④ не за job-каталогом, а як trust + правовий обов'язок (зберігаємо телефон/селфі).
- **Інфраструктурні/правові екрани (групи G–H зі sitemap) лягають у наявну навігацію, без нової вкладки.** Вхід existing-user і правова згода — у контейнері **Онбординг** (вхід — альтернативна точка старту до таб-бару). Підтримка, Умови/Політика, Відновлення доступу, Налаштування приватності, Заблоковані — секції вкладки **④ Профіль** (логічний дім керування собою і даними). Це навмисно: вони не породжені job, тож не заслуговують власної вкладки — живуть як службові пункти кабінету.
- **Екрани — зі sitemap.** Додані «Мої заявки (надіслані)», «Керування акаунтом» і нові групи **G–H** (вхід, правові, підтримка, lifecycle оголошення, підтвердження заселення, форма скарги, приватність, заблоковані) внесено й у [sitemap.md](./sitemap.md), тож навігація і мапа лишаються синхронні. «Онбординг», «Таб-бар» і назви вкладок — навігаційні контейнери, не екрани.
- **Безпека/приватність** — інваріанти (видимість person-card, утримання на платформі, скарга/блок, re-verify фото) у [sitemap.md «Рішення безпеки і приватності»](./sitemap.md#рішення-безпеки-і-приватності-до-wireframes); впливатимуть на стани екранів на етапі wireframes.
