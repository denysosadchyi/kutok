## Сутності

> Інвентаризація головних об'єктів, з якими людина має справу, щоб закрити свої jobs.
> Кожна сутність народжена конкретним job. Поля — лише ті, що job вимагає або яких вимагає скоуп MVP (CLAUDE.md).

### Карта зв'язків (огляд)

```
            ┌─────────────────┐
            │  Профіль людини │◄────── Відгук (про кого / від кого)
            └────────┬────────┘
                     │ володіє
          ┌──────────┴───────────┐
          │                      │
   роль: господар          роль: шукач
          │                      │ (профіль сам = «картка людини»)
          ▼                      │
   ┌──────────────┐              │
   │  Оголошення  │              │
   │  (кімната)   │              │
   └──────┬───────┘              │
          │                      │
          │  обидва потрапляють у │
          ▼                      ▼
        ┌────────────────────────┐
        │   Пошук / фільтр       │  (вибірка над каталогом)
        └────────────────────────┘
                     │
   хтось натискає «Хочу познайомитись»
                     ▼
            ┌─────────────────┐
            │ Інтерес/заявка  │  ── взаємний? ──►  розблоковує
            └─────────────────┘                        │
                                                        ▼
                                                  ┌──────────┐
                                                  │   Чат    │
                                                  └──────────┘
```

### 1. Профіль людини

Центральний об'єкт продукту. Одна людина = один профіль; роль (шукач / господар / обидві) — не окремий об'єкт, а контекст використання того самого профілю. Для **шукача** профіль сам по собі є «карткою людини» у стрічці (CLAUDE.md: два типи картки — кімната і людина).

**Поля / частини:**
- Ім'я, вік
- Фото (звірене із селфі — не просто завантажене)
- «Про себе»
- Робота / рід занять `[?]` (тригер довіри в personas, але не в явному MVP-списку полів)
- Звички: курить / не курить, тварини, режим `[?]` (тригери Олени; для фільтра й RJ4)
- **Статус верифікації / бейдж з розшифровкою** — структурована частина: `телефон ✓ / фото ✓ / документ —`. Центральний механізм довіри; завжди живе всередині профілю, не окремо.
- Повнота профілю (completeness) — як видимий сигнал `[?]` (jtbd позначає окремий UI-індикатор кандидатом на виліт; наповненість лишається сигналом у будь-якому разі)
- Відгуки про людину (агрегуються сюди, див. сутність «Відгук»)

**Породжуючі jobs:** Main Job (впевнитися, що людина реальна), [RJ2](./research/jtbd.md) (переконатися до контакту), [RJ4](./research/jtbd.md) (прийти на зустріч підготовленим), [SJ1](./research/jtbd.md) (виглядати серйозним кандидатом), [SJ2](./research/jtbd.md) (довіра новачку без «хто мене знає»).

**Зв'язки:** володіє **Оголошенням** (у ролі господаря) · є об'єктом і автором **Інтересу/заявки** · учасник **Чату** · об'єкт і автор **Відгуку**.

---

### 2. Оголошення (кімната)

Об'єкт сторони пропозиції — те, що розміщує господар. Окремий від профілю: один господар може мати кімнату, але об'єкт опису житла ≠ опис людини (research: ставка Кутка — профіль *людини*, але кімната теж потрібна як предмет угоди; CLAUDE.md MVP: «Сторінка оголошення кімнати»).

**Поля / частини:**
- Фото кімнати/житла
- Опис
- Умови: ціна, район
- Правила співжиття: курити / тварини / режим `[?]` (дзеркало звичок, для фільтра й сумісності)
- Дата заселення `[?]`
- Посилання на профіль господаря

**Породжуючі jobs:** [RJ1](./research/jtbd.md) (відсіяти нерелевантне — кімната є об'єктом фільтрації), Main Job з боку шукача (знайти безпечну кімнату). Розміщення кімнати обслуговує jobs Олени-господаря (personas: «розміщує кімнату»).

**Зв'язки:** належить **Профілю** (господар) · потрапляє у **Пошук/фільтр** · є ціллю **Інтересу/заявки**.

---

### 3. Інтерес / заявка («Хочу познайомитись»)

Об'єкт-намір. Фіксує однобічний інтерес; коли інтерес взаємний — служить ключем, що розблоковує чат. Це сам по собі сигнал довіри (research, патерн №4: незнайомець не може написати напряму).

**Поля / частини:**
- Автор (профіль, що ініціював)
- Ціль (профіль людини або оголошення кімнати)
- Статус: надіслано / взаємно / відхилено `[?]`
- Дата `[?]`

**Породжуючий job:** [RJ3](./research/jtbd.md) (почати розмову тільки якщо інтерес взаємний). Для господаря — це «кандидати», яких він переглядає (personas: Олена «переглядає кандидатів»).

**Зв'язки:** з'єднує два **Профілі** (або профіль ↔ оголошення) · при взаємності породжує **Чат**.

---

### 4. Чат

Простір розмови, що відкривається лише після взаємного інтересу. Окремий від «Інтересу»: заявка — це гейт (намір), чат — наповнення (розмова). MVP свідомо базовий — лише текст (CLAUDE.md).

**Поля / частини:**
- Учасники (два профілі)
- Повідомлення (текст)
- Прив'язка до заявки, що його розблокувала `[?]`

**Породжуючий job:** [RJ3](./research/jtbd.md) (почати розмову за взаємного інтересу). Дотично [EJ1](./research/jtbd.md)/[EJ2](./research/jtbd.md) — захищений канал знижує тривогу першого контакту.

**Зв'язки:** належить парі **Профілів** · розблокований **Інтересом/заявкою**.

---

### 5. Відгук

Сліди досвіду спільного проживання — peer-reference. Народжується після проживання, прив'язується до профілю людини як сигнал репутації.

> Примітка скоупу: jtbd позначає відгуки **кандидатом на post-MVP** (cold start — на старті відгуків немає → порожній стан). Job [RJ5](./research/jtbd.md) реальний, тому сутність лишається в основному списку, але з міткою «функція вмикається після першої хвилі матчів».

**Поля / частини:**
- Автор (профіль — попередній господар або сусід)
- Об'єкт (профіль, про кого відгук)
- Текст
- Роль автора відносно об'єкта: господар / сусід `[?]`
- Дата; вікно подачі 14 днів (research R5, бенчмарк Couchsurfing)
- Двосторонність — обидві сторони лишають, публікація одночасна `[?]` (Airbnb-патерн; для UA-валідації)

**Породжуючий job:** [RJ5](./research/jtbd.md) (підтвердити свою надійність для наступного разу).

**Зв'язки:** прив'язаний до двох **Профілів** (автор → об'єкт) · агрегується у **Профіль** як сигнал довіри.

---

### 6. Пошук / фільтр

Не власний об'єкт людини, а **вибірка (view) над каталогом** оголошень і профілів. Включив у основний список, бо його прямо породжує job, але за природою це механізм навігації над сутностями 1–2, а не річ, якою людина «володіє».

**Поля / частини (критерії фільтра):**
- Район
- Ціна
- Стать
- (completeness / верифікація як сигнал на картці результату — не критерій, а атрибут елемента)

**Породжуючий job:** [RJ1](./research/jtbd.md) (швидко відсіяти очевидно нерелевантне).

**Зв'язки:** оперує над **Оголошеннями** і **Профілями (шукачів)**.

## Під питанням

> Об'єкти, які jobs прямо не вимагають, або вимагає лише гіпотеза. Не тягнути в дизайн без перевірки.

| Кандидат-об'єкт | Чому під питанням |
|---|---|
| **Збережені / обране** (закладки оголошень, збережений пошук) | Жоден job не вимагає. Зручність, не прогрес людини. Не вигадувати без потреби. |
| **Сповіщення** (push) | Свідомо поза MVP (CLAUDE.md). Жодного job-носія. |
| **Скоринг сумісності** | Гіпотеза в jtbd, свідомо post-MVP (research, патерн №3 — ламається на cold start). Job не підтверджений на UA. |
| **Спільні знайомі / соц-граф** `[?]` | Job у jtbd позначений гіпотезою («дуже бажана фіча», але поведінкових даних немає). На вибірку до інтерв'ю. |
| **Completeness score як окремий об'єкт** | jtbd: кандидат на виліт; сам по собі не закриває жодного критичного (3) job. Лишаю як *поле-сигнал* профілю, не як самостійну сутність. |
| **Бейдж довіри як окрема сутність** | Не існує поза профілем — це структурована частина «Профілю» (статус верифікації), не самостійний об'єкт. Винесено сюди свідомо, щоб не плодити фантомну сутність. |
