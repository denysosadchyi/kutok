# User Flows — Куток

> Чернетка. Кожен flow — під заголовком job ([jtbd.md](./research/jtbd.md)).
> Вузли `[у квадратних]` = екрани зі [sitemap.md](./sitemap.md), розділ «Екрани». Рішення `{ромб?}`.
> Показано не лише happy-path: стани **empty / loading / error** — окремими вузлами, і обидва кінці — успіх **і** тупики.
>
> **Легенда станів і кінців** (єдина для всіх діаграм):
> - помаранчевий — проміжний стан (empty / loading / error)
> - червоний — тупик, де людина застрягає
> - зелений — успішний кінець
>
> **Звірка зі sitemap:** усі вузли-екрани існують у sitemap.md. Нових екранів не введено — «Хочу познайомитись» і «фільтр» лишаються діями/станами екранів (як зафіксовано в sitemap), не окремими екранами.
>
> **Периметр (свідоме обмеження).** Намальовано: MAIN, RJ1, RJ2, RJ3 (бік Ані-шукача) + бік **господаря (E, Олена)**. Поки без діаграм: RJ4 (готовність до зустрічі — покриває екран «Профіль іншої людини» + лінк із чату), RJ5 (відгуки — post-MVP; якір заселення в [sitemap.md](./sitemap.md)), SJ1/SJ2, EJ1/EJ2. Безпекові гілки (скарга/блок, re-verify фото, приватність) — рішення у [sitemap.md «Рішення безпеки і приватності»](./sitemap.md#рішення-безпеки-і-приватності-до-wireframes).

## MAIN JOB — безпечно знайти, з ким і де жити

> *Коли я шукаю людину для спільного проживання серед незнайомих, я хочу бути впевненою, що вона реальна, щоб прийняти рішення без страху пошкодувати.*
> Primary — Аня (шукач). Повний наскрізний flow.

```mermaid
flowchart TD
    Start(["Аня заходить уперше"]) --> Create["Створення профілю"]
    Create --> Feed["Стрічка / каталог: кімнати + люди, з фільтром"]

    Feed --> Q1{"Є результати?"}
    Q1 -->|ні| Empty["Стан: нічого не знайшлось"]
    Empty --> Loosen["Дія: послабити / змінити фільтр"]
    Loosen --> Feed
    Q1 -->|так| View["Профіль іншої людини / Оголошення кімнати"]

    View --> Q2{"Профіль вселяє довіру?"}
    Q2 -->|ні| Feed
    Q2 -->|так| Q3{"Я верифікована?"}

    Q3 -->|ні| Verify["Верифікація: телефон + селфі"]
    Verify --> Q4{"Верифікація успішна?"}
    Q4 -->|ні| VerErr["Стан: збій (SMS не дійшла / код невірний / фото не збіглось / нема дозволу на камеру)"]
    VerErr --> Q4b{"Кількість невдач?"}
    Q4b -->|менше N| Verify
    Q4b -->|N і більше| Esc["Ескалація: ручна перевірка / підтримка; стрічку видно й без верифікації"]
    Esc --> Feed
    Q4 -->|так| Send
    Q3 -->|так| Send["Дія: «Хочу познайомитись» → Інтерес"]

    Send --> Qs{"Заявку надіслано?"}
    Qs -->|збій| NetErr["Стан: помилка мережі"]
    NetErr -->|повторити| Send
    NetErr -->|зберегти і вийти| Feed
    Qs -->|так| Wait["Стан: чекаємо відповідь (loading)"]

    Wait --> Q5{"Відповіли на заявку?"}
    Q5 -->|тиша| Stuck["Тупик: відповіді немає"]
    Q5 -->|відхилили| Rej["Стан: заявку відхилено"]
    Q5 -->|так, взаємно| Chat["Чат (текст)"]
    Rej --> Feed
    Stuck --> Feed

    Chat --> Meet["Домовились про зустріч (поза застосунком)"]
    Meet --> Success(["Успіх: знайшла, з ким і де жити"])

    classDef state fill:#fff3e0,stroke:#e8a33d,color:#5b4a2f;
    classDef stuck fill:#fde4e1,stroke:#d2553f,color:#5b2a22;
    classDef ok fill:#e7f3ea,stroke:#4a9d6a,color:#234a33;
    class Empty,VerErr,NetErr,Wait,Rej,Esc state;
    class Stuck stuck;
    class Success ok;
```

**Точки зриву:** порожня стрічка (empty) → послаблення фільтра або відвал (cold-start окремо — див. RJ1); збій верифікації → після N невдач **ескалація** (не вічна петля), стрічку видно й без неї; тиша/відмова на заявку — найчастіше. Успіх відбувається **поза застосунком** (зустріч), тож продукт відповідає лише за безпечний шлях до неї — і за утримання розмови до зустрічі ([sitemap: anti-Viber](./sitemap.md#рішення-безпеки-і-приватності-до-wireframes)).

## RJ1 — знайти серед багатьох тих, хто базово підходить

> *Коли я починаю шукати серед десятків профілів, я хочу швидко відсіяти тих, хто точно не підходить.*
> Фокус: фільтр і стан порожнього результату.

```mermaid
flowchart TD
    A["Стрічка / каталог: кімнати + люди, з фільтром"] --> B["Дія: фільтр (район, ціна, стать) — панель Стрічки"]
    B --> L["Стан: оновлюємо результати (loading)"]
    L --> C{"Є результати?"}

    C -->|ні| W{"Чому порожньо?"}
    W -->|фільтр завузький| E["Стан: під фільтр нічого"]
    W -->|база порожня cold-start| ZI["Стан: ще немає пропозицій"]

    E --> F{"Послабити фільтр?"}
    F -->|так| B
    F -->|ні| Reset["Скинути фільтр і дивитися все"]
    Reset --> A

    ZI --> ZN["CTA: сусідні райони / «повідомити, коли з'являться»"]
    ZN --> A

    C -->|так| G["Переглядає картки"]
    G --> H{"Є хоч одна підходяща?"}
    H -->|ні| B
    H -->|так| I["Профіль іншої людини / Оголошення кімнати"]
    I --> Ok(["Далі → RJ2: оцінити довіру"])

    classDef state fill:#fff3e0,stroke:#e8a33d,color:#5b4a2f;
    classDef ok fill:#e7f3ea,stroke:#4a9d6a,color:#234a33;
    class E,L,ZI,ZN,Reset state;
    class Ok ok;
```

**Точки зриву:** два різні «порожньо» — *фільтр завузький* (→ послабити / скинути) і *cold-start, база порожня* (→ сусідні райони / «повідомити, коли з'являться»). Жоден більше не термінальний тупик; на старті домінує саме cold-start, і послаблення фільтра тут не зарадить.

## RJ2 — переконатися, що людина реальна, до будь-якого контакту

> *Коли я натрапила на профіль, що здається підходящим, я хочу переконатися, що це реальна людина, щоб не ризикувати наодинці з незнайомцем.*
> Ядро MVP. Фокус: trust layer — бейдж із розшифровкою, повнота, відгуки.

```mermaid
flowchart TD
    A["Стрічка / каталог: кімнати + люди, з фільтром"] --> B["Профіль іншої людини"]
    B --> C{"Є бейдж верифікації?"}
    C -->|ні| Stuck1["Тупик: непідтверджений — не ризикує"]
    C -->|так| D["Дивиться розшифровку: телефон ✓ / фото ✓ / документ —"]

    D --> E{"Профіль заповнений?"}
    E -->|ні, порожній| Stuck2["Сумнів: повертається у Стрічку"]
    E -->|так| F{"Є відгуки?"}

    F -->|ні| G["Стан: ще немає відгуків (cold start)"]
    F -->|так| H["Читає відгуки про людину"]

    G --> I{"Довіри достатньо без відгуків?"}
    H --> I
    I -->|ні| Stuck2
    I -->|так| Trust(["Готова до контакту → RJ3"])

    Stuck1 --> A
    Stuck2 --> A

    classDef state fill:#fff3e0,stroke:#e8a33d,color:#5b4a2f;
    classDef stuck fill:#fde4e1,stroke:#d2553f,color:#5b2a22;
    classDef ok fill:#e7f3ea,stroke:#4a9d6a,color:#234a33;
    class G state;
    class Stuck1,Stuck2 stuck;
    class Trust ok;
```

**Точки зриву:** відсутній бейдж = миттєвий тупик (research R1/R2); порожній профіль = сумнів; **cold start відгуків** — empty-стан, що на старті стосуватиметься майже всіх профілів (тому відгуки = post-MVP).

## RJ3 — почати розмову тільки якщо інтерес взаємний

> *Коли я вирішила написати першою, я хочу знати, що людина навпроти теж відкрита, щоб не відчувати, що вторгаюсь.*
> Ядро MVP. Двосторонній gate: чат відкривається лише при взаємності.

```mermaid
flowchart TD
    A["Профіль іншої людини / Оголошення кімнати"] --> B["Дія: «Хочу познайомитись» → Інтерес"]
    B --> C{"Заявку надіслано?"}
    C -->|збій| Err["Стан: помилка мережі"]
    Err -->|повторити| B
    Err -->|зберегти і вийти| My
    C -->|так| My["Мої заявки: надіслані / очікують (loading)"]

    My -->|передумала / «дзвіночки»| Wd["Дія: відкликати заявку"]
    Wd --> A
    My --> Sig["In-app сигнал на вкладці Заявки (лічильник, не push)"]

    Sig --> E{"Вхідні в іншої сторони"}
    E -->|ще немає заявок| EmptyIn["Стан: порожньо — ще немає кандидатів (cold-start)"]
    E -->|є| F{"Інша сторона теж цікавиться?"}

    F -->|ні, відхилила| Rej["Стан: заявку відхилено"]
    F -->|тиша| Silent["Стан: без відповіді (заявка лишається активною)"]
    F -->|так, взаємно| G["Чат (текст) — розблоковано"]

    Rej --> Back["Список діалогів"]
    Silent --> Back
    EmptyIn --> Back
    Back -->|шукати далі| A

    G --> H{"Є повідомлення?"}
    H -->|ще немає| GEmpty["Стан: ще немає повідомлень — почніть розмову"]
    H -->|так| Ok(["Успіх: розмова почалась"])
    GEmpty --> Ok

    classDef state fill:#fff3e0,stroke:#e8a33d,color:#5b4a2f;
    classDef ok fill:#e7f3ea,stroke:#4a9d6a,color:#234a33;
    class Err,My,Rej,Silent,EmptyIn,GEmpty,Sig,Wd state;
    class Ok ok;
```

**Точки зриву:** відмова й тиша — два різні негативні кінці (відхилення видиме, тиша — ні, але заявка лишається активною); порожні вхідні й порожній чат на старті — стани за замовчуванням (cold-start); без **in-app сигналу** обидві сторони мусили б навмання перевідкривати вкладки → відчуття «мертвого» продукту (research, патерн №4 «коли ламається»). Можливість **відкликати** заявку — дія безпеки.

## E (бік господаря) — підселити, не впустивши «не ту людину»

> *Коли я підселяю в свій дім, я хочу відфільтрувати кандидатів до зустрічі й переконатися, що людина реальна, щоб не впустити проблему.*
> Олена. Сторона пропозиції: майже не користується стрічкою — її «пошук» це перегляд кандидатів, що прийшли на оголошення.

```mermaid
flowchart TD
    Start(["Олена заходить"]) --> Onb["Онбординг: профіль + верифікація"]
    Onb --> Create["Створення оголошення (кімната): фото, опис, умови"]
    Create --> Pub{"Опубліковано?"}
    Pub -->|збій аплоуду фото| UpErr["Стан: помилка завантаження"]
    UpErr -->|повторити| Create
    Pub -->|так| Live["Оголошення активне"]

    Live --> Inbox{"Є вхідні заявки?"}
    Inbox -->|ще немає| EmptyC["Стан: ще немає кандидатів (cold-start) — порада «як зробити оголошення помітнішим»"]
    EmptyC --> Wait["Чекає (in-app сигнал на нову заявку)"]
    Wait --> Inbox
    Inbox -->|є| Cand["Вхідні заявки / кандидати"]

    Cand --> View["Профіль кандидата: бейдж з розшифровкою, повнота, відгуки"]
    View --> Q{"Реальний і підходить?"}
    Q -->|ні / дзвіночки| Decline["Відхилити / поскаржитись-заблокувати"]
    Decline --> Cand
    Q -->|так| Accept["Прийняти заявку → взаємно"]
    Accept --> Chat["Чат (текст) — розблоковано"]
    Chat --> Meet["Зустріч (поза застосунком)"]
    Meet --> Ok(["Успіх: знайшла сусіда, якому довіряє"])

    classDef state fill:#fff3e0,stroke:#e8a33d,color:#5b4a2f;
    classDef ok fill:#e7f3ea,stroke:#4a9d6a,color:#234a33;
    class UpErr,EmptyC,Wait state;
    class Ok ok;
```

**Точки зриву:** аплоуд фото (error/loading); **порожні вхідні на cold-start** — стан за замовчуванням на старті, тому потрібна порада, а не просто порожнеча; «реальність» кандидата тримається на тому ж бейджі з розшифровкою (RJ2 з боку господаря). `[?]` Відкрите: якщо primary — Олена, її головний шлях (E → кандидати) не має шорткату-стрічки — перевірити на інтерв'ю.
